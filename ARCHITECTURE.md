## 1. Обзор

Агент — это чат-интерфейс для мониторинга IT-инфраструктуры. Пользователь пишет вопрос на естественном русском языке («что с report3?», «как биллинг?», «покажи CPU на db1»), агент автономно:

1. разбирает намерение через LLM (GigaChat),
2. делает нужные запросы в Zabbix API,
3. формирует контекст из полученных данных,
4. генерирует человекочитаемый ответ через LLM,
5. стримит результат на клиент по SSE.

Агент **не угадывает** данные — вся информация берётся исключительно из Zabbix. Если сервер не найден, агент честно сообщает об этом.

## 4. Общий pipeline обработки запроса

```
POST /chat
│
├── want_expert=true? ──────────────────────────────► Ветка 6.1 (экспертный разбор)
│
├── prev_graph_candidates? ─────────────────────────► Ветка 6.2 (выбор метрики)
│
├── Фаза 1: LLM анализирует текущее сообщение
│   └── возвращает: (ident, ident_type, graph_query, wants_server_list, wants_metric_list)
│
├── Нет нового ident? → подставить prev_server / prev_system из контекста
│
├── ident_type == "system"? ────────────────────────► Ветка 6.3 (поиск системы)
│
├── ident есть, тип != system
│   └── _zabbix_lookup_smart(ident) ────────────────► Ветка 6.4 (поиск сервера)
│       │
│       ├── graph_query? ──────────────────────────► Ветка 6.5 (график)
│       ├── wants_metric_list? ────────────────────► Ветка 6.6 (список метрик)
│       └── иначе ─────────────────────────────────► Ветка 6.7 (статус сервера)
│
└── нет ident, нет намерений ──────────────────────► Ветка 6.8 (разговор)
```

Каждая ветка в конце генерирует SSE-поток с ответом LLM.

## 5. Фаза 1 — анализ запроса LLM

**Функция:** `_llm_analyze_request(user_msg, history, model, custom_prompt)`

Анализирует **только текущее сообщение** (не всю историю — это принципиально, чтобы не переносить контекст туда, где его быть не должно).

Возвращает кортеж:
```
(identifier, ident_type, graph_query, wants_server_list, wants_metric_list)
```

### Типы идентификаторов

| Тип | Описание | Пример |
|---|---|---|
| `hostname` | Имя хоста (буквы + цифра на конце) | `report3`, `db1`, `bill2` |
| `ip` | IPv4-адрес | `10.10.1.101` |
| `ci` | Номер КЭ (учётная запись оборудования) | `CI0101289001` |
| `system` | Название группы/подсистемы | `биллинг`, `авторизация` |
| `none` | Идентификатор не найден | — |

LLM обязана нормализовать падежи: «биллинга» → «биллинг», «авторизацию» → «авторизация».

### Три попытки извлечения (каскадный fallback)

Анализ устойчив к сбоям модели — три попытки с деградацией:

**Попытка 1 — function calling (основная)**

Отправляет запрос с принудительным вызовом функции `analyze_request`:
```json
{
  "function_call": {"name": "analyze_request"},
  "functions": [{ "name": "analyze_request", "parameters": {...} }]
}
```
GigaChat возвращает JSON в поле `function_call.arguments`. Этот путь самый надёжный — модель обязана вернуть структурированный JSON.

**Попытка 2 — JSON в тексте (fallback)**

Если function calling не сработал (ошибка сети, неожиданный ответ), добавляет в системный промпт суффикс с инструкцией вернуть plain JSON. Ищет JSON-объект в ответе через регулярку `\{[^{}]+\}`.

**Попытка 3 — минималистичный промпт без истории**

Если обе попытки провалились — отправляет упрощённый промпт только с текущим сообщением, просит вернуть только `{identifier, type}`. Минимизирует шансы на отказ.

Если все три попытки не дали результат — возвращает `(None, None, None, False, False)` и диалог переходит в разговорный режим (ветка 6.8).

## 6. Ветки действий

### 6.1 Экспертный разбор по требованию (fast path)

**Триггер:** `want_expert=true` в теле запроса.

Вызывается фронтендом при нажатии кнопки «Экспертный разбор» на уже отображённом ответе. Не проходит через фазу анализа — сразу работает с `prev_server` или `prev_system` из тела запроса.

**Для сервера:**
1. `_zabbix_lookup(prev_server)` — получает hostid
2. `_zabbix_get_triggers(hostid)` — все триггеры
3. Фильтрует только активные (`value == "1"`)
4. Если активных нет — сообщает об этом, завершает
5. Иначе — отправляет в `_gigachat_stream` с промптом `PROMPT_EXPERT`, стримит `expert_chunk`-события

**Для системы:**
1. `_zabbix_search_groups(prev_system)` — ищет группу
2. `_zabbix_get_system_data(groupid)` — хосты + их активные триггеры
3. Фильтрует хосты с проблемами
4. Если проблем нет — сообщает, завершает
5. Иначе — стримит `expert_chunk` с промптом `PROMPT_EXPERT_SYSTEM`

### 6.2 Выбор метрики из кандидатов (fast path)

**Триггер:** `prev_graph_candidates` (список) + `prev_graph_hostid` непустые.

Возникает когда на предыдущем шаге LLM нашла несколько метрик и попросила пользователя уточнить. Пользователь отвечает — «первую», «2», «cpu utilization» и т.д.

**Логика разрешения** (`_resolve_graph_candidate`):
1. **Числительные/порядковые слова** через `_ORDINAL_MAP` — словарь из русских порядковых слов и цифр в индексы (0–7). Поиск через `\b` word boundary.
2. **Подстрока имени** — имя кандидата содержится в сообщении или наоборот.
3. **Пересечение слов** — хотя бы одно слово длиннее 2 символов из сообщения встречается в имени метрики.

Если кандидат найден — немедленно запрашивает историю метрики и рисует график (без обращения к LLM для анализа).

### 6.3 Поиск системы (группы серверов)

**Триггер:** `ident_type == "system"`.

**Поиск группы** (`_zabbix_search_groups`):
- Сначала точное совпадение по имени (`filter`)
- Если не найдено — wildcard-поиск, max 5 результатов

**Три исхода:**

**а) Группа не найдена:**
- Запрашивает все группы с мониторируемыми хостами (`_zabbix_all_group_names`)
- Формирует контекст `_CTX_SYSTEM_NOT_FOUND` со списком всех доступных групп
- Сохраняет список в `system_candidates` (для следующего хода)

**б) Найдено несколько совпадений:**
- Формирует контекст `_CTX_SYSTEM_CLARIFY` с нумерованным списком вариантов
- Сохраняет кандидатов в `system_candidates`
- На следующем ходу, если пользователь называет один из них, `ident_from_candidates=True` переопределяет идентификатор

**в) Найдена ровно одна группа:**
- `_zabbix_get_system_data(groupid)` — одним запросом получает всех хостов группы (лимит 500) и все их активные триггеры, объединяет по `hostid`
- Если `wants_server_list=true` → контекст `_CTX_SYSTEM_HOST_LIST` (список всех хостов, первые 50, остальные суммаризируются)
- Иначе → контекст `_CTX_SYSTEM_FOUND` только с проблемными хостами
- Если есть проблемные хосты и это свежий запрос (`fresh_ident=true`) — **автоматически** запускает экспертный разбор через `PROMPT_EXPERT_SYSTEM` (стримит `expert_chunk` до основного ответа)

### 6.4 Поиск конкретного сервера

**Функция:** `_zabbix_lookup_smart(ident)` → `(name, info, fuzzy_match)`

**Двухуровневый поиск:**

1. `_zabbix_lookup(ident)` — строгий поиск по трём полям:
   - `host.get filter={"host": [ident]}` — точное имя хоста
   - `hostinterface.get filter={"ip": [ident]}` → `host.get hostids=[...]` — поиск по IP
   - `host.get searchInventory={"alias": ident}` — поиск по номеру КЭ из инвентаря

2. Если строгий поиск не дал результата — wildcard-поиск:
   - `host.get search={"host": "*ident*"} searchWildcardsEnabled=True limit=1`
   - `fuzzy_match=True` — фронтенд покажет уведомление «ближайший совпадающий»

Каждый хост нормализуется через `_norm_host`: извлекается главный IP-интерфейс (`main == "1"`), данные инвентаря (alias = КЭ, os), первая группа (берётся только последний сегмент пути через `/`).

**Особый случай:** если пришёл запрос на график/метрики, но сервер не найден по текущему идентификатору, и в контексте есть `prev_server` — используется `prev_server` (пользователь мог написать «покажи CPU» не называя сервер повторно).

### 6.5 График метрики

**Триггер:** `graph_query` непустой и `effective_hostid` известен.

1. `_zabbix_get_all_items(hostid)` — все включённые метрики хоста (лимит 200, сортировка по имени)
2. Фильтрует только числовые типы: `value_type in (0, 3)` (float и uint64)
3. `_llm_pick_metric(user_msg, items, model)` — LLM выбирает нужные метрики из полного списка:
   - Формирует нумерованный список с единицами измерения
   - Просит LLM ответить номерами через запятую
   - Парсит числа из ответа, возвращает до 4 совпадений

**Три исхода:**

**а) Метрики не найдены** → текстовый ответ с предложением уточнить название

**б) Найдена одна метрика:**
- `_zabbix_get_item_history_timed(itemid, value_type, hours=3)` — история за 3 часа (лимит 500 точек)
- `_make_item_chart(name, unit, points)` → PNG в base64
- SSE-событие `graph` с изображением

**в) Найдено несколько метрик:**
- SSE-событие `meta` с `graph_candidates` — список объектов `{itemid, name, units, value_type}`
- Фронтенд рисует кликабельный список
- На следующем ходу — ветка 6.2 (fast path)

Метрику из списка метрик (ветка 6.6) можно также кликнуть — тогда вызывается `POST /graph` напрямую, без LLM.

### 6.6 Список метрик сервера

**Триггер:** `wants_metric_list=true` и `effective_hostid` известен.

1. `_zabbix_get_all_items(hostid)` — все включённые метрики (лимит 200)
2. SSE-событие `metric_list` с полным списком `{itemid, name, units, value_type}`
3. Фронтенд рендерит прокручиваемую таблицу; числовые метрики кликабельны (открывают график через `POST /graph`)
4. Нечисловые (`value_type not in [0, 3]`) отображаются полупрозрачно, клик не работает

### 6.7 Статус сервера (основной поток)

1. `_zabbix_get_triggers(hostid)` — все триггеры хоста, отсортированные по приоритету (DESC)
2. Для каждого триггера вычисляется `ago_min` — сколько минут назад последнее изменение (только для активных)
3. Приоритеты маппируются: `{0: INFO, 1: INFO, 2: LOW, 3: MEDIUM, 4: HIGH, 5: HIGH}`
4. Активные (`value == "1"`) отделяются от нормальных

**Если есть активные триггеры и это свежий запрос (`fresh_ident=true`):**
- Автоматически запускается `_gigachat_stream` с `PROMPT_EXPERT` — SRE-анализ с матрицей приоритетов
- Стримится до основного ответа как `expert_chunk`-события

**Контекст для основного ответа** (`_CTX_FOUND`):
- Данные сервера: name, ip, ci, os, group
- Число активных и общих триггеров
- Строки активных триггеров с severity и временем
- Инструкция LLM: не перечислять нормальные триггеры, дать резюме в 2-3 предложения

Если хост не найден — используется `_CTX_NOT_FOUND` с идентификатором.  
Если идентификатор не найден — `_CTX_NO_IDENT`.

### 6.8 Разговорный режим (без идентификатора)

**Триггер:** нет свежего идентификатора, нет намерений (graph/list).

Агент отвечает в контексте последнего объекта разговора:

- Если есть `prev_system` — добавляет блок `[КОНТЕКСТ ДИАЛОГА]` с названием системы
- Если есть `prev_server` — добавляет блок с именем сервера
- Иначе — добавляет `_CTX_NO_IDENT` с просьбой уточнить

Поддерживает до 8 предыдущих сообщений истории. Поддерживает прикреплённые изображения (только для моделей из `_VISION_MODELS`).

## 7. Клиент Zabbix

### Сессия и автоперелогин

Zabbix JSON-RPC требует токен сессии, который получается через `user.login`. Токен хранится в глобальной переменной `_zbx_token`. 

`asyncio.Lock` (`_zbx_lock`) защищает от race condition при параллельных запросах: если несколько корутин одновременно обнаружат `_zbx_token is None`, только одна выполнит логин.

При получении ошибки `"Not authorized"` токен сбрасывается (`_zbx_token = None`) и запрос автоматически повторяется — уже с новым токеном. Это покрывает истечение сессии Zabbix.

Каждый запрос получает уникальный `id` (инкрементальный `_zbx_rid`) — требование JSON-RPC протокола.

### Методы Zabbix API, используемые агентом

| Метод | Назначение |
|---|---|
| `user.login` | Получение токена сессии |
| `host.get` | Поиск хостов по имени, hostid, wildcard, инвентарю |
| `hostinterface.get` | Поиск хоста по IP |
| `trigger.get` | Триггеры хоста или группы хостов |
| `item.get` | Метрики хоста (все или по подстроке имени) |
| `history.get` | История значений метрики за период |
| `hostgroup.get` | Поиск групп по имени (точный + wildcard) |

## 8. Клиент GigaChat

### SSL

GigaChat API использует сертификат Sber CA, который не входит в стандартные хранилища. Создаётся кастомный SSL-контекст с отключённой верификацией (`CERT_NONE`, `check_hostname=False`). Применяется ко всем запросам к GigaChat.

### Токен OAuth и кэширование

`_get_token()` — async-функция, возвращает Bearer-токен. Реализует кэш:
- Токен хранится в `_token_cache = {"token": None, "expires_at": 0.0}`
- Перед каждым запросом проверяет: `now < expires_at - 60` (60 секунд запас)
- Если токен валиден — возвращает кэшированный без HTTP-запроса
- При обновлении: POST на `_AUTH_URL` с `Basic {_AUTH_KEY}` и `scope=GIGACHAT_API_PERS`
- `expires_at` из ответа в миллисекундах, делится на 1000; дефолт +30 минут

### Три режима обращения к API

**`_gigachat(messages, model)`** — обычный запрос, возвращает строку с ответом. Используется в фазе анализа и выборе метрики.

**`_gigachat_fn_call(messages, functions, model, force_fn)`** — function calling. При `force_fn` устанавливает `"function_call": {"name": force_fn}`, иначе `"auto"`. Возвращает `(content, function_call_dict)`. Используется исключительно для анализа запроса (ветка 5).

**`_gigachat_stream(messages, model)`** — async generator, читает SSE от GigaChat (`text/event-stream`), вычленяет `delta.content` из каждого чанка, пропускает `[DONE]` и ошибочные строки. Timeout 120 секунд. Используется для всех пользовательских ответов и экспертного разбора.

### Допустимые модели

```python
_ALLOWED_MODELS = {"GigaChat-2-Max", "GigaChat-2-Pro", "GigaChat-2"}
_VISION_MODELS  = {"GigaChat-2-Max", "GigaChat-2-Pro"}
```

Если пришедшая модель не в списке — автоматически используется `_MODEL` (дефолт из env).

## 9. Рендеринг графиков

**Функция:** `_make_item_chart(title, unit, points)` → base64 PNG

Рисует временной ряд средствами Pillow без внешних зависимостей типа matplotlib.

### Параметры холста

- Размер: **720 × 260 px**
- Отступы: L=72, R=24, T=32, B=36 (левый большой — для подписей Y)
- Тема: тёмная (Catppuccin-style)

### Цветовая схема

| Элемент | RGB |
|---|---|
| Фон | `(20, 22, 34)` |
| Сетка | `(45, 48, 65)` |
| Линия | `(166, 227, 161)` — зелёная |
| Заливка | `(166, 227, 161, 50)` — полупрозрачная |
| Текст | `(160, 165, 195)` |
| Заголовок | `(210, 215, 235)` |

### Алгоритм

1. **Ось Y:** 6 горизонтальных линий сетки, равномерно. Диапазон расширяется на 12% сверху и снизу (`v_pad`). Если все значения одинаковы — искусственно расширяется на `max(1.0, |v| * 0.1 + 1)`.
2. **Ось X:** вертикальные линии каждые 30 минут от первой до последней точки. Подписи в UTC (`HH:MM`).
3. **Заливка:** полигон от нижнего края до линии графика и обратно вниз.
4. **Линия:** сегменты между соседними точками, толщина 2px.
5. **Заголовок:** строка `"{title}   avg {avg}   max {max}"` над графиком.
6. Конвертация RGBA → RGB, сохранение в BytesIO, base64-кодирование.

Если точек меньше 2 — возвращает пустую строку (фронтенд покажет сообщение «нет данных»).

## 10. SSE-протокол стриминга

Все события имеют формат:
```
data: {"type": "<тип>", ...}\n\n
```

| Тип | Данные | Когда |
|---|---|---|
| `meta` | Структурированные данные о сервере/системе, флаги | В начале ответа, до текста |
| `chunk` | `{"text": "..."}` | Фрагменты основного ответа LLM |
| `expert_chunk` | `{"text": "..."}` | Фрагменты экспертного разбора |
| `graph` | `{"chart": "<base64>", "title": "...", "unit": "..."}` | Когда построен график |
| `metric_list` | `{"host": "...", "total": N, "items": [...]}` | Список метрик |
| `error` | `{"text": "..."}` | Ошибка обработки |
| `done` | `{"reply": "<полный текст>"}` | Завершение потока |

### Структура `meta`

```json
{
  "type": "meta",
  "server_name": "report3",
  "server_info": {"hostid": "...", "ip": "...", "ci": "...", "os": "...", "group": "..."},
  "system_name": null,
  "system_hosts": [],
  "triggers": [{"title": "...", "severity": "HIGH", "active": true, "ago_min": 45}],
  "n_ok": 12,
  "chart": null,
  "fuzzy_match": false,
  "fuzzy_query": null,
  "graph_candidates": null,
  "graph_hostid": "1234",
  "system_candidates": [],
  "system_not_found": false
}
```

Фронтенд обновляет весь контекст (prev_server, prev_system, graph_candidates и т.д.) из первого же `meta`-события, не дожидаясь `done`.

## 11. Управление контекстом диалога

Контекст **не хранится на сервере** — он передаётся клиентом в каждом запросе. Сервер stateless.

### Поля контекста (тело POST /chat)

| Поле | Тип | Назначение |
|---|---|---|
| `message` | string | Текущее сообщение пользователя |
| `history` | array | Последние 8 пар `{role, content}` |
| `server_name` | string\|null | Последний упомянутый сервер |
| `system_name` | string\|null | Последняя упомянутая система |
| `system_candidates` | array | Кандидаты при неоднозначном запросе системы |
| `prev_graph_candidates` | array | Кандидаты метрик при неоднозначном графике |
| `prev_graph_hostid` | string\|null | hostid для которого были кандидаты |
| `want_expert` | bool | Запрос экспертного разбора |
| `file_ids` | array | ID загруженных через `/upload` файлов |
| `model` | string | Выбранная модель |
| `prompts` | object\|null | Кастомные промпты из редактора UI (null = дефолты) |

### Логика подстановки идентификатора

```
fresh_ident = LLM нашла идентификатор в текущем сообщении

if not fresh_ident and prev_server:
    ident = prev_server, type = "hostname"

if not fresh_ident and prev_system:
    ident = prev_system, type = "system"

if prev_candidates and ident:
    # проверить, совпадает ли ident с одним из кандидатов
    # (подстрока в любую сторону)
    → ident_from_candidates = True, ident_type = "system"
```

`fresh_ident` влияет на автозапуск экспертного разбора: разбор запускается только при свежем идентификаторе, не при продолжении разговора о том же сервере.

## 12. Система промптов

Все промпты имеют дефолты в коде и могут быть переопределены через UI или через поле `prompts` в запросе.

| Ключ | Назначение |
|---|---|
| `extract` | Фаза 1 — инструкция для анализа и извлечения идентификатора |
| `base` | Роль агента, правила поведения, тон |
| `no_ident` | Когда сервер/система не упомянуты |
| `not_found` | Хост не найден в Zabbix. Плейсхолдер: `{ident}` |
| `found` | Данные найденного хоста. Плейсхолдеры: `{name}` `{ip}` `{ci}` `{os}` `{group}` `{n_active}` `{n_total}` `{n_ok}` `{trigger_lines}` |
| `expert` | SRE-анализ активных проблем сервера с матрицей приоритетов |
| `host_metrics` | Список метрик сервера. Плейсхолдеры: `{name}` `{ip}` `{n_items}` `{item_lines}` |
| `system_found` | Состояние системы. Плейсхолдеры: `{group}` `{n_hosts}` `{n_problem_hosts}` `{n_active_total}` `{host_lines}` |
| `system_host_list` | Список серверов системы. Плейсхолдеры: `{group}` `{n_hosts}` `{host_lines}` |
| `system_nf` | Система не найдена. Плейсхолдеры: `{ident}` `{suggestions}` |
| `system_clarify` | Несколько совпадений. Плейсхолдеры: `{query}` `{n}` `{candidates}` |
| `expert_system` | SRE-анализ активных проблем всей системы |

Подстановка плейсхолдеров через `_safe_format` — устойчива к отсутствующим ключам (не бросает исключение, оставляет `{key}` как есть).

## 13. Загрузка файлов / изображений

**Endpoint:** `POST /upload` (multipart/form-data, поле `file`)

1. Читает файл из multipart-запроса
2. Отправляет в GigaChat Files API (`/api/v1/files`) с `purpose=general`
3. Возвращает `{"file_id": "...", "filename": "..."}`

Полученный `file_id` фронтенд включает в следующий запрос `/chat` в поле `file_ids`. Бэкенд добавляет в сообщение пользователя:
```json
{"role": "user", "content": "...", "attachments": [{"type": "image", "id": "file_id"}]}
```

Изображения поддерживаются только моделями из `_VISION_MODELS` (Pro и Max). Для Lite изображение молча игнорируется.

## 14. Фронтенд

Реализован на vanilla JS без фреймворков. Единственный HTML-файл `templates/index.html`.

### Компоненты интерфейса

**Шапка:** логотип, название, селектор модели (Pro/Max/Lite), кнопка редактора промптов.

**Редактор промптов (⚙):** выдвижная панель с textarea для каждого промпта. Загружает дефолты с `GET /prompt`. При изменении показывает бейдж «изменён». «Применить» сохраняет в `customs`, «Сбросить» возвращает дефолты. Кастомные промпты отправляются в каждый запрос `/chat`.

**Чат-область:** flex-колонка, автоскролл вниз. Пользовательские пузыри — справа (синий градиент), ответы агента — слева (тёмный фон).

**Серверная карточка** — появляется под ответом при `meta.server_info`: имя, IP, КЭ, ОС, система.

**Системная карточка** — появляется при `meta.system_name`: название группы, число хостов, число проблемных.

**Список триггеров** — отображается при `meta.triggers`. Цвет точки: HIGH = красный, MEDIUM = оранжевый, LOW = жёлтый. Для систем показывается `host_name`.

**Кнопка «Показать график CPU»** — появляется под триггерами при наличии активных проблем у сервера. При нажатии отправляет «покажи график нагрузки CPU» в чат.

**График** — `<img>` с base64 PNG в карточке с зелёной рамкой.

**Кандидаты графика** — кликабельный список метрик. Клик → отправить индекс в чат (через `sendMessage(String(i + 1))`).

**Список метрик** — прокручиваемая таблица (max 260px). Числовые метрики кликабельны — вызывают `fetchGraphItem`, который делает `POST /graph` напрямую и вставляет график в текущий пузырь без нового сообщения.

**Экспертный разбор** — сворачиваемый блок. Появляется автоматически при наличии активных проблем. Кнопка «Экспертный разбор» позволяет вызвать по требованию.

**Индикатор набора** — три анимированных точки во время ожидания ответа.

**Прикрепление файла** — кнопка 📎 открывает `<input type="file" accept="image/*">`. Файл сразу загружается на `/upload`, `file_id` буферизуется до отправки следующего сообщения. Предпросмотр отображается над полем ввода.

### SSE-клиент на стороне браузера

Используется `fetch` + `ReadableStream` вместо `EventSource` — это позволяет отправлять POST с телом. Async generator `readSSE(response)` читает поток побайтово, склеивает в буфер, разбивает по `\n\n`, парсит `data:` строки.

## 15. Переменные окружения

Загружаются из `.env` через `python-dotenv` в `main.py` до импорта `zabbix_bot`.

| Переменная | Обязательная | Дефолт | Описание |
|---|---|---|---|
| `GIGACHAT_AUTH_KEY` | **да** | — | Base64-строка `client_id:secret` для OAuth |
| `ZABBIX_URL` | **да** | — | URL JSON-RPC: `http://host/api_jsonrpc.php` |
| `ZABBIX_PASS` | **да** | — | Пароль пользователя Zabbix |
| `ZABBIX_USER` | нет | `Admin` | Имя пользователя Zabbix |
| `GIGACHAT_MODEL` | нет | `GigaChat-2-Pro` | Модель по умолчанию |
| `GIGACHAT_SCOPE` | нет | `GIGACHAT_API_PERS` | OAuth scope |
| `GIGACHAT_AUTH_URL` | нет | стандартный Sber | Endpoint получения токена |

| `GIGACHAT_FILES_URL` | нет | стандартный Sber | Endpoint загрузки файлов |

Обязательные переменные читаются через `os.environ[...]` — приложение упадёт при старте с `KeyError` если переменная не задана.
